# Аудит дизайна и визуальной системы платформы «Прозрачная цена»

**Дата:** 19.07.2026
**Метод:** независимый аудит «с чистого листа» — без опоры на внутреннюю историю, прошлые аудиты и текущие дизайн-решения. Платформа оценивается против канонов крупных мировых продуктов (Stripe, Linear, Vercel, Airbnb, Shopify Polaris, IBM Carbon, Bloomberg-класс рабочих столов).
**Рамки:** весь фронтенд (`apps/web`): публичный контур, 12 ролевых кабинетов, ~225 страниц, ~450 компонентов, 39 глобальных CSS-файлов.
**Принцип:** ни одна функция и ни одна опция не удаляется. Аудит касается только формы: как это выглядит, как читается, как управляется.

---

## 1. Резюме (TL;DR)

**Продуктовая основа — сильная. Визуальная система — в состоянии «археологических слоёв».**

У платформы есть то, что большинство B2B-продуктов не имеют даже на зрелой стадии: осмысленная тёмная тема с собственным характером (глубокий зелёный, а не «серый как у всех»), семантические цвета по доменам сделки (деньги / документы / логистика / споры), command palette, skip-links и ARIA-дисциплина, локализация ru/en/zh, «хребет сделки» как навигационная идея. Это уровень намерений дорогих мировых платформ.

Но реализация — это **пять поколений дизайна, лежащих друг на друге** (v7 → v7r → v8 → v9 → transaction-ux), склеенных 39 глобальными CSS-файлами с именами `emergency`, `fix`, `final`, `stable`, `hotfix`. Каждый следующий слой не заменял предыдущий, а перекрывал его через `!important`. Итог измерим:

| Метрика | Факт | Норма мировых платформ |
|---|---|---|
| Уникальных hex-цветов в CSS | **366** | 25–40 токенов |
| Hex-цветов, зашитых прямо в TSX | **315** | 0 |
| `!important` в глобальных стилях | **1 323** | ≈ 0 (единицы, только для a11y-оверрайдов) |
| Inline `style={{…}}` в компонентах | **5 546** вхождений | единицы (динамические координаты) |
| Глобальных CSS-файлов | **39** (11 400 строк) | 1 токен-файл + модули |
| Разных мобильных брейкпоинтов | **15+** (360, 380, 390, 420, 430, 560, 600, 640, 720, 721, 767, 768, 820, 980, 1080…) | 3–4 |
| Максимальный z-index | **2 147 483 601** (`platform-v7-public-assistant-mobile-fix.css:48`) | шкала из ~8 уровней |
| Компонентных слоёв | 5 (121 + 257 + 59 + 7 + 2 файла) | 1 система + фичи |
| Компонентов в «официальной» дизайн-системе v8 | **2** из ~450 | покрытие 90%+ |

Диагноз: **платформа не «плохо задизайнена» — она задизайнена пять раз, и все пять версий работают одновременно.** Пользователь чувствует это как микронесогласованность: чуть разные кнопки, чуть разные отступы, чуть разные оттенки на соседних экранах, скачки вёрстки на мобильном. Команда чувствует это как страх трогать CSS.

Ремонт возможен без единой потери функции: 90% работы — консолидация уже существующих хороших решений, а не изобретение новых.

## 2. Что уже сделано на мировом уровне (сохранить как фундамент)

Аудит обязан фиксировать не только болезни. Эти вещи — актив, их нельзя потерять при рефакторинге:

1. **Токен-слой `styles/theme.css`** — грамотная двухуровневая архитектура: примитивы `--p7-color-*` → семантические алиасы `--pc-*`. Есть радиусы, спейсинг 4/8, тени, шрифтовая переменная. Это готовый скелет дизайн-системы — он просто не является законом.
2. **Доменные семантические цвета** — `--p7-color-money`, `--p7-color-evidence`, `--p7-color-logistics`, `--p7-color-document`, `--p7-color-dispute`. Это редкое и очень сильное решение для сделочной платформы: цвет кодирует *домен события*, а не абстрактную «важность». Ни у Stripe, ни у Linear такого словаря нет — это ваше конкурентное ноу-хау. Развернуть на полную (см. §6.3).
3. **AppShellV4 (`components/v7r/`)** — современный шелл: lucide-иконки (не эмодзи!), command palette, ролевая навигация, переключатель темы, CSS-модули. Это правильный «единственный дом» для всех кабинетов.
4. **Доступность как привычка**: skip-link на публичной главной, `aria-labelledby` на секциях, `prefers-reduced-motion` учтён в 36 файлах, `focus-visible` в 16, семантический `<details>/<summary>` вместо JS-аккордеона. Выше среднего по индустрии.
5. **Локализация ru/en/zh** с отдельным CJK-типографическим слоем.
6. **Управляемость дизайна как процесс**: `design-governance-v8.json` (39 версий!), CI-гейт `design:v8:guard`, список «мигрированных» файлов. Механизм принуждения уже построен — его нужно только докормить.
7. **Идея «пути сделки»** на публичной главной (10 этапов, перспективы 12 ролей) — сюжетная подача продукта, как у Stripe «follow the payment». Это правильный нарратив.

## 3. Диагноз по слоям

### 3.1 Архитектура стилей: «каскад заплаток»

39 глобальных CSS-файлов подключаются вразнобой (41 `import '@/styles/…'` разбросан по страницам и компонентам). Имена файлов — честная история болезни:

```
platform-v7-emergency-mobile.css
platform-v7-public-hero-offset-fix.css
platform-v7-open-login-final.css
platform-v7-public-entry-stable.css        ← 206 × !important
platform-v7-unified-modal-fullscreen.css   ← 238 × !important
design-fixes.css                            ← 79 × !important
mobile-polish.css                           ← 95 × !important
```

Каждый «fix»-файл — это война специфичности с предыдущим слоем, выигранная через `!important`. Побочный эффект: следующее изменение возможно только ещё более сильным `!important` — система необратимо дорожает в поддержке. Порядок подключения файлов стал недокументированным API: переставь два импорта — и поедет вёрстка логина.

**Антипаттерн года** — стилизация по *значению inline-атрибута* (`styles/mobile-polish.css:93`):

```css
:root:not([data-theme='light']) .pc-main div[style*='#F8FAFB'] {
  background: var(--p7-color-surface) !important;
}
```

Тёмная тема «ловит» светлые inline-цвета селектором по подстроке hex-кода. Это значит: inline-стили в компонентах официально признаны неуправляемыми, и CSS охотится на них по сигнатурам, как антивирус. Любой новый светлый hex в TSX (`#F8FAFC` вместо `#F8FAFB`) молча пробивает тёмную тему.

### 3.2 Inline-стили: параллельная нелегальная дизайн-система

5 546 вхождений `style={{…}}` в компонентах + 575 в страницах, 315 уникальных hex-цветов прямо в TSX (лидеры: `PrivacyPortalPanel.tsx` — 48, `BuyerFavoritesPanel.tsx` — 46, `GrainReleaseCockpit.tsx` — 44). Фактически большая часть интерфейса задизайнена «на месте», мимо токенов. Отсюда:
- тема не может гарантированно применяться (см. 3.1);
- одинаковые по смыслу элементы на разных экранах выглядят по-разному;
- невозможно глобально поменять ни один параметр (радиус, отступ, оттенок).

### 3.3 Палитра: 366 оттенков вместо 40 токенов

При наличии готового токен-слоя в `theme.css` в CSS живёт 366 уникальных hex-значений (+315 в TSX, частично пересекаются). Это не «богатая палитра», это шум: десятки почти неразличимых серо-зелёных, конкурирующие оттенки успеха/опасности, случайные синие. Контраст текста на этих случайных парах никто не проверял — гарантий WCAG AA нет нигде, кроме токенизированных зон.

### 3.4 Типографика: четыре шрифтовых стека

Одновременно объявлены: `"Inter Variable", Inter, Arial…`, `Inter, "SF Pro Text"…`, `Inter, ui-sans-serif, system-ui… !important` и `Arial, "Helvetica Neue" !important`. Плюс два моноширинных (`SFMono/Consolas` и `JetBrains Mono`). Файл `typography-tokens.css` существует (272 строки) — но рядом лежит `platform-v7-public-typography.css` (496 строк, 73 × `!important`), который его пересиливает. Шкала размеров не едина: заголовки на соседних экранах отличаются на 1–2 px без причины.

### 3.5 Сетка и брейкпоинты: 15 порогов адаптива

360, 380, 390, 420, 430, 560, 600, 640, 720, 721, 767, 768, 820, 980, 1080 px. Пары 720/721 и 767/768 означают, что разные файлы считают «планшетом» разные вещи → на ширинах 721–768 px разные части экрана живут в разных режимах одновременно. У зрелых систем порогов 3–4 и они являются токенами.

### 3.6 Слоистость (z-index): война до Int32

Шкалы нет. Значения: 1, 2, 3, 5, 30, 40, 200, 300, 2600, 3500, 5000… и чемпион — `2147483601 !important` (максимум int32 + 2) у мобильного ассистента. Это гарантированные баги: тосты под модалками, дропдауны под шапкой, ассистент поверх системных диалогов. И это же ломает доступность: визуальный порядок слоёв перестаёт соответствовать порядку фокуса.

### 3.7 Компонентный слой: пять параллельных вселенных

| Слой | Файлов | Статус |
|---|---|---|
| `components/` (корневой, v7-эпоха) | 121 | живой, inline-стили |
| `components/platform-v7/` | 257 | живой, основной объём, inline-стили |
| `components/v7r/` (AppShellV4 и др.) | 59 | живой, самый современный |
| `components/transaction-ux/` | 7 | живой, под governance |
| `packages/design-system-v8/` | **2** | «официальная» система |

Дизайн-система, объявленная законом (`design-governance-v8.json`), покрывает 2 компонента из ~450. Таблицы, формы, кнопки, модалки, бейджи реализованы по 3–5 раз в разных слоях. Для пользователя это выглядит как «платформа собрана из разных продуктов»; для команды — как невозможность исправить что-либо в одном месте.

### 3.8 Информационная архитектура: 110 маршрутов без иерархии

Внутри `/platform-v7` — 110 top-level директорий. Среди них дубли и почти-дубли: `auction` и `auctions`, `counterparty` и `counterparties`, `lot`, `lots` и `buyer-lot`, `deal-flow`, `deals`, `deal-acceptance`, `deal-drafts`, `deal-documents-basis`, `deal-logistics`. Функционально это богатство; навигационно — плоское поле из сотни дверей. Ролевая навигация в AppShellV4 частично спасает, но: (а) не все страницы живут в шелле, (б) публичный контур, кабинеты и staff-зона используют три разных навигационных парадигмы.

### 3.9 Темизация: тёмная — принудительная, светлая — на честном слове

`theme.css` объявляет `color-scheme: dark` на `:root` безусловно. Светлая тема существует как набор исключений-селекторов (`:root:not([data-theme='light'])…`) и «версии по умолчанию» в `PlatformThemeSync`. `prefers-color-scheme` не используется вообще — системная настройка пользователя игнорируется. Упоминается `data-theme='high-contrast'` — задел хороший, но без систематической реализации.

### 3.10 Мобильный опыт: спасён вручную, держится на страхе

Файлы `platform-v7-emergency-mobile.css`, `platform-v7-login-mobile-stability.css`, `platform-v7-public-assistant-mobile-fix.css` (84 × `!important`), `platform-v7-mobile-notification-safe.css` говорят сами за себя: мобильная вёрстка не проектировалась mobile-first, а чинится вдогонку. Safe-area (вырезы iPhone) обрабатывается в четырёх местах по-разному. При этом ключевые полевые роли платформы — водитель, сюрвейер, элеватор — живут именно в телефоне, часто на солнце и со слабой связью. Для них мобильный экран — это не адаптив, это основной продукт.

### 3.11 Публичный контур: три поколения «product experience» одновременно

Главная страница импортирует восемь CSS-файлов, среди которых `…experience-v3.css` (1 734 строки), `…v3-refinement.css`, `…v4.css`, `…v5.css` — все четыре активны разом, плюс `<style>` прямо в JSX (`app/platform-v7/page.tsx:122`). Контент-нарратив главной — отличный; его стилевая основа — палимпсест.

## 4. Целевая концепция: «Стеклянный элеватор»

Формулировка «северной звезды», против которой принимаются все дизайн-решения:

> **Платформа — это один непрерывный рентген сделки.** Пользователь любой роли в любой момент видит: где сделка сейчас, что подтверждено доказательствами, что требует его действия. Всё остальное — фон.

Из этого следуют пять законов интерфейса:

1. **Хребет сделки — главный навигационный объект.** Не меню из 110 пунктов, а горизонтальная лента этапов (условия → допуск → логистика → приёмка → качество → документы → деньги → закрытие), одинаковая на публичной главной, в кабинете и на мобильном. Каждый экран платформы — это «вид на хребет» под углом конкретной роли. (Основа уже есть: `TOUR_STAGES`, `deal-spine-panel`, `deal-lifecycle-lane` — их нужно возвести в закон, а не держать как виджеты.)
2. **Цвет = домен, а не украшение.** Существующий словарь (деньги — синий, документы — голубой, доказательства — фиолетовый, логистика — сиреневый, спор — розовый, успех/бренд — зелёный) применяется строго и только к семантике. Всё несемантическое — нейтральная поверхность. Пользователь через неделю читает экран периферийным зрением.
3. **Одно действие на экран.** У каждой роли на каждом этапе есть ровно одно главное действие (уже есть `next-step-bar`, `NextActionCard` — правильные зачатки). Оно всегда в одном месте, всегда одного вида. Остальные опции — не удаляются, а убираются во второй визуальный ранг (меню «ещё», раскрываемые секции).
4. **Доказательство — визуальный первоклассный объект.** Подписанный документ, результат пробы, платёж — отображаются как «пломбы»: единый компонент-штамп с хешем, временем, автором. Это и есть бренд «Прозрачной цены».
5. **Тишина по умолчанию.** Ни градиентов, ни свечения, ни теней там, где нет смысла. Дорогие платформы выглядят дорого за счёт дисциплины пустоты, а не за счёт эффектов.

## 5. Дизайн-система 2.0 (спецификация)

Ни одного нового артефакта «с нуля» — всё это консолидация существующего.

### 5.1 Токены (единственный источник: `packages/design-tokens`)

- **Цвет:** ~36 токенов. 10 нейтральных ступеней (тёплый серо-зелёный от `#07110F` до `#F4F7F5`), бренд (3 ступени), 4 статуса × 2 (цвет + soft-фон), 7 доменных пар из §2.2. Все пары «текст/фон» проходят WCAG AA 4.5:1, проверка — скриптом в CI.
- **Типографика:** один стек `"Inter Variable", Inter, system-ui, sans-serif` + один моно `"JetBrains Mono", ui-monospace, monospace`. Шкала: 12 / 13 / 14 (базовый) / 16 / 18 / 22 / 28 / 36, интерлиньяж 1.5 для текста, 1.2 для заголовков, `tabular-nums` для всех чисел и денег (обязательно для торговой платформы!).
- **Пространство:** только шкала 4-8-12-16-24-32-48-64.
- **Брейкпоинты — ровно четыре:** 480 (телефон), 768 (планшет), 1080 (десктоп), 1440 (широкий). Как токены, с едиными именами.
- **Z-index — именованная шкала из 8 уровней:** base 0, raised 10, sticky 100, header 200, dropdown 300, overlay 400, modal 500, toast 600, assistant 650, debug 700. Числовые z-index вне шкалы запрещены линтером.
- **Motion:** 120 ms (микро), 200 ms (панели), 320 ms (сцены); единая кривая `cubic-bezier(0.2, 0, 0, 1)`; всё отключается через `prefers-reduced-motion` централизованно, одним правилом.

### 5.2 Темы

Три темы как равноправные наборы токенов: **тёмная** (текущая, основная), **светлая** (полноценная, не селекторы-исключения), **полевая** (high-contrast: чёрный/белый/жёлтый, min размер шрифта 16, min тач-цель 48 px — для водителей и работы на солнце). По умолчанию — `prefers-color-scheme`, переключатель в шелле сохраняет выбор. Хак `div[style*='#F8FAFB']` удаляется вместе с причиной (inline-стилями).

### 5.3 Ядро компонентов (~30 примитивов в `design-system-v8`)

Button (4 варианта × 3 размера), Input/Select/Textarea/DatePicker, Badge/StatusChip (единственная реализация статусов на платформе), Card/Panel, Table (+ мобильная карточная проекция), Modal/Drawer/Popover (один менеджер слоёв), Toast, Tabs, Breadcrumbs, EmptyState (уже есть), Skeleton (уже есть), KPI-tile, Timeline/Spine, **EvidenceSeal** (пломба из §4.4), FileDrop, Stepper, CommandPalette (уже есть). Каждый примитив: CSS-модуль, только токены, story-пример, светлая+тёмная+полевая темы из коробки.

### 5.4 Правила-запреты (CI, расширение `design:v8:guard`)

1. Новые `!important` — запрещены (allowlist только для a11y-оверрайдов).
2. Hex/rgb-литералы в TSX и CSS — запрещены; только `var(--…)`.
3. `style={{…}}` — только для действительно динамики (координаты карт, проценты прогресса), проверка по allowlist свойств.
4. Числовой z-index — запрещён, только токены шкалы.
5. Новые файлы в `styles/*.css` — запрещены; только CSS-модули и токен-файл.
6. Медиазапросы — только через токенизированные брейкпоинты.

Дашборд метрик прямо в CI: количество hex-литералов, `!important`, inline-стилей — цифры должны только падать (ratchet-механика: PR не может увеличить счётчик).

## 6. План миграции (без остановки продукта, без потери функций)

**Фаза 0 — заморозка ямы (1 неделя).** Включить ratchet-гейты из §5.4 в режиме «не хуже». Ни одна строка функционала не меняется.

**Фаза 1 — фундамент (2–3 недели).** Консолидировать токены: `theme.css` + `typography-tokens.css` + v9-переменные → единый `packages/design-tokens`. Свести брейкпоинты к четырём, z-index к шкале. Собрать 30 примитивов §5.3 (большинство — «канонизация» лучшей из существующих реализаций, не написание новой).

**Фаза 2 — один шелл (2–4 недели).** AppShellV4 становится единственным контейнером всех 12 кабинетов и staff-зоны. Дубли навигации (`role-navigation`, `mobile-nav`, `module-hub`, page-frame-варианты) сводятся к нему. Хребет сделки закрепляется как постоянный элемент шелла на страницах сделки.

**Фаза 3 — зачистка страниц волнами (6–10 недель, параллелится).** Порядок по трафику и боли: (1) логин/регистрация и публичная главная — схлопнуть experience-v3/v3r/v4/v5 в один файл; (2) кабинеты покупатель/продавец/оператор; (3) сделки и деньги; (4) полевые роли — сразу в полевой теме; (5) staff/админ. Правило волны: страница переводится на примитивы, её inline-стили и hex-литералы уходят, соответствующий `*-fix.css` **удаляется целиком**. Функциональность фиксируется скриншот- и e2e-тестами до/после — ни одна опция не пропадает, меняется только оболочка.

**Фаза 4 — полировка мирового уровня (постоянно).** Микро-анимации по motion-токенам, пустые состояния с иллюстрациями, плотность интерфейса, звуковая тишина.

**Критерий финиша, измеримый:** hex-литералы 366→0 (только токены), `!important` 1323→<20, глобальные CSS 39→2 (tokens + reset), inline-стили 5546→<100 (только динамика), брейкпоинты 15→4, z-index max 2147483601→650.

## 7. От себя: фишки и ноу-хау, которых нет у конкурентов

То, что просили «добавить от себя» — идеи поверх ремонта, каждая опирается на уже существующие данные платформы:

1. **«Паспорт сделки» — трекинг как у посылки, доверие как у нотариуса.** Одна страница-урл на сделку, где хребет этапов сшит с пломбами доказательств. Шарится контрагенту, банку, арбитру — каждый видит свой уровень детализации. Печатная/PDF-версия одной кнопкой — это ежедневная реальность зернового бизнеса, и её никто на рынке не делает красиво.
2. **Генеративный «отпечаток сделки».** Из хеша цепочки доказательств рендерится уникальный визуальный узор (стилизованный колос/поле) — на карточке сделки, в PDF, в шапке паспорта. Красиво, брендово и функционально: подделку видно глазами. Дешёво в реализации (SVG из хеша), выглядит как ноу-хау уровня больших финтехов.
3. **Полевой режим — как отдельный продукт, а не адаптив.** Для водителя/сюрвейера: экраны из одного действия, кнопки во весь палец, контраст под прямое солнце, офлайн-очередь с честным индикатором «что уйдёт при появлении связи» (основа уже есть: `field-mobile-hub`, `weak-connection-mode-panel`, `offline-conflict-panel` — их нужно возвести в первоклассный режим с собственной темой).
4. **Режим плотности для операторов.** Оператор и банк живут в таблицах — переключатель comfortable/compact (32 px строки, моно-цифры) в шелле. Стоит копейки при токенизированном спейсинге, воспринимается как забота уровня Bloomberg.
5. **Command palette как главный вход в 110 маршрутов.** ⌘K уже есть — докормить его: поиск по сделкам/лотам/контрагентам/действиям, «перейти к этапу сделки», недавние. Для платформы со 110 разделами палитра — не фича, а решение проблемы навигации.
6. **«Пульс рынка» в шелле.** Тонкая строка тикера (цены базисов, индекс активности) — использует существующие `market-news-rail`/`market-center-workspace`. Даёт ощущение живой площадки с первого взгляда, как у торговых терминалов.
7. **Тихие уведомления.** Группировка по сделке + дайджест «за время вашего отсутствия» вместо ленты из 40 строк. Меньше тревоги — больше доверия.
8. **Онбординг «сыграй сделку за 90 секунд».** Публичный тур по 10 этапам уже построен — довести его до интерактивной песочницы с демо-данными, где посетитель нажимает реальные кнопки. Лучший продавец платформы — сама платформа.

## 8. Итог

Платформе не нужен редизайн в смысле «нарисовать заново». Ей нужна **консолидация: пять дизайнов → один**, с уже существующими токенами в роли закона и уже существующим CI-гейтом в роли полиции. Продуктовые идеи (хребет сделки, доменные цвета, доказательства-пломбы) — мирового уровня; они просто погребены под слоями заплаток. Снять слои — и платформа будет выглядеть так же дорого, как устроена внутри.
