# Каноническая сделка — конституция проекта

Дата принятия: 16.07.2026. Статус: **действует**.
Цель проекта: **промышленная система федерального уровня**, готовая к внешним
подключениям и показу партнёрам и банку. Не демо и не пилот: каждый экран и
каждый шов делаются сразу в боевом качестве (см. §0).
Этот документ определяет, что входит в ядро. Всё, чего нет в этом документе,
не разрабатывается и не показывается, пока ядро не готово.

## 0. Планка качества (промышленная, не демонстрационная)

1. **Никаких демо-обходов в продуктовом пути.** Верификация организации — только
   через решение оператора в его консоли (SQL-обход из LOCAL_RUNBOOK — строго
   локальная отладка). Ни один статус не меняется мимо API и журнала.
2. **Аудит и доказуемость.** Каждое действие ядра пишет журнал: кто, когда,
   основание, результат. Существующие тесты на мультитенантность и утечки —
   обязательный гейт, они не ослабляются ради скорости.
3. **Отказоустойчивость видима.** Недоступность внешней системы (банк, ФГИС, ЭДО)
   переводит операцию в явное состояние ожидания с ответственным и сроком —
   никогда в ошибку на весь экран и никогда в тихую потерю.
4. **Дизайн — официальный, ясный, функциональный**, уровня зрелых мировых
   финансовых платформ: плотность данных × строгая иерархия × деловой русский
   язык. Эталон: `design-prototype/lk-prototype.html` (v3),
   `DESIGN_PLAYBOOK_cabinets_2026-07-16.md`. Без развлекательности, без эмодзи
   в рабочих экранах, без жаргона; таблицы с табличными цифрами, статусы чипами,
   у каждой строки — следующий шаг.
5. **Производительность и масштаб.** Ядро проектируется под тысячи организаций и
   десятки тысяч сделок: пагинация и лимиты на каждом списке, индексы под
   основные выборки, никаких выборок «всё сразу».

## 1. Сценарий

Одна внебиржевая зерновая сделка от партии до закрытия. Цена определяется
на торгах (аукцион) либо прямым предложением — обе ветки входят в ядро:

| № | Этап | Кто действует | Что фиксируется |
|---|------|--------------|-----------------|
| 1 | Партия и лот | продавец | партия, качество по документам продавца, публикация лота |
| 2 | Торги / предложение | покупатели ↔ продавец | аукцион: окно торгов, ставки, лучшая ставка, итог; либо прямое предложение и согласие. Цена, объём, базис, версия условий |
| 3 | Сделка и резерв | покупатель, банк | сделка создана из итога торгов, резерв заявлен, банк подтверждает |
| 4 | Перевозка | логист, водитель | рейс, привязка к сделке, прибытие |
| 5 | Приёмка | элеватор | фактический вес, расхождение, акт |
| 6 | Качество | лаборатория | проба, протокол |
| 7 | Документы | все стороны | комплект оснований для расчёта |
| 8 | Расчёт | банк, оператор | выплата по фактическому весу, удержание по спорной части |
| 9 | Спор (ветка) | арбитр, оператор | удержание, решение, доказательный пакет |
| 10 | Закрытие | система | сделка закрыта, журнал полон |

Требования к аукциону (этап 2): серверное время окна — единственный источник
правды; ставки атомарны и журналируются; итог торгов создаёт сделку без ручного
переноса данных; проигравшие участники видят честный итог. В API уже есть
аукционный контур и CI-гейт `auction-atomic-acceptance` — ядро строится на нём.

**Модель видимости лотов (решение владельца продукта, 16.07.2026).**
Лот виден **всем допущенным покупателям платформы**, но в **обезличенном** виде:
культура, класс, объём, регион/базис, качество, стартовая цена и окно торгов —
без наименования продавца, точного адреса и любых контактов, чтобы исключить
сделку в обход платформы. Деанонимизация — поэтапная и журналируемая:
наименования сторон раскрываются только после создания сделки из итога торгов,
прямые контакты — на стадии исполнения, где это операционно необходимо
(существующие механизмы `ContactRevealGate` / anti-bypass используются как
основа). Текущая tenant-изоляция это не поддерживает (см. PHASE1_BACKLOG №1) —
требуется публичная витрина лотов поверх tenant-модели.

## 2. Роли ядра

**В ядре (6):** продавец, покупатель, оператор, элеватор, логист/водитель, банк-наблюдатель.
**Вне ядра до готовности ядра:** сюрвейер, лаборатория как отдельный кабинет (протокол
качества оформляет оператор/элеватор), инвестор, комплаенс, руководитель, арбитр
(спор в ядре обрабатывает оператор).

## 3. Экраны ядра (целевой набор, ≈15)

- Вход, регистрация организации.
- Кабинет роли = Обзор (KPI + очередь решений + события) — один шаблон на все роли.
- Торги: список лотов с окнами торгов → карточка лота (ставки, лучшая ставка,
  серверный отсчёт окна, итог) — для продавца и покупателя.
- Сделки (таблица) → Карточка сделки (факты, вкладки: хронология / документы / деньги / качество).
- Экран решения (вариант + последствие + подтверждение) — один шаблон.
- Деньги (таблица позиций), Документы (таблица статусов).
- Полевой экран приёмки (элеватор) и рейса (водитель, мобильный).
- Консоль оператора: верификация организаций, очереди задач + песочница банка (см. §5).

Эталон визуала и поведения: `design-prototype/lk-prototype.html` (v3) и
`DESIGN_PLAYBOOK_cabinets_2026-07-16.md`.

## 4. Данные

Каждый экран ядра читает и пишет только через API (`@pc/api`). Зашитые в JSX
суммы, сделки и статусы запрещены; демо-данные живут в сидере тестового тенанта.

## 5. Интеграционные швы (готовность к подключению)

Для каждого внешнего партнёра — адаптер с контрактом (OpenAPI), песочница и
точка замены на боевого клиента:

1. **Банк — целевой партнёр АО «Россельхозбанк».** Контракт шва: запрос
   подтверждения резерва → вебхук статуса → основание выплаты → вебхук исполнения →
   выписка. Песочница: оператор играет роль банка в своей консоли. Примитивы
   держим банк-нейтральными (резерв/удержание/выплата/выписка), специфика РСХБ
   (эскроу-счёт или спецсчёт, формат авторизации, их стандарт Open API) ложится
   в адаптер после получения технических условий. Контекст: у РСХБ есть
   экосистема «Своё Фермерство» (агромаркетплейс, 21 тыс.+ поставщиков),
   партнёрские API-программы (прецедент «Своё Жильё») и эскроу-продукты —
   позиционируем платформу как исполнительный контур сделки, дополняющий их
   витрину, а не конкурирующий с ней.
2. **ФГИС «Зерно» / СДИЗ:** проверка статуса СДИЗ, фиксация погашения. Песочница:
   ручное подтверждение оператором с журналом.
3. **ЭДО (подписание):** отправка документа на подпись → вебхук подписи. Песочница:
   имитация подписи с водяным знаком «тест».

«Платформа готова» = сквозной сценарий проходит на песочницах без ручных правок
данных, а подключение реального партнёра — замена адаптера и ключей.

## 6. Что заморожено

- Маршруты вне §3 (150+ страниц platform-v7, дерево platform-v7r, pc-public-entry) —
  не развиваются; по мере переноса ядра выводятся из эксплуатации.
- Новые CSS-файлы и цветовые литералы — запрещены гейтом `design:freeze:guard`.
- Новые панели/виджеты в кабинеты — только если они часть экрана из §3.

## 7. Порядок изменений

Изменение этого документа — отдельный коммит с обоснованием. Добавить в сценарий
новую возможность можно только удалив или завершив что-то равного объёма.
