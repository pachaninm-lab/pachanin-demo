# Стандарт дизайна и UX платформы «Прозрачная цена» (v1)

**Статус:** обязательный стандарт. Дополняет `AUDIT_design-visual_2026-07-19.md`: аудит отвечает «что не так», этот документ — «как обязано быть». Каждая волна миграции из аудита принимается только по критериям этого стандарта.

## 0. Требования владельца продукта (зафиксированы, не пересматриваются)

1. **Понятность для каждого.** Любой пользователь — включая человека без опыта работы с компьютерными системами, уставшего, спешащего, со слабым зрением или на дешёвом телефоне в поле — выполняет свою задачу без обучения и без посторонней помощи.
2. **Взрослый промышленный вид.** Платформа выглядит и ощущается как федеральная отраслевая система, а не как стартап-демо: строго, документально, надёжно.
3. **Полнота функций.** Ни одна функция или опция, заложенная в коде, не удаляется и не прячется безвозвратно. Упрощение = переранжирование, а не ампутация.
4. **Ноль «демо-ощущения».** Служебные, тестовые и симуляционные элементы не видны рабочим ролям.

Важно: требование «промышленного вида» реализуется дизайном и не отменяет регламент честности из `AGENTS.md` (запрещённые заявления о зрелости интеграций остаются запрещёнными). Система выглядит серьёзной потому, что она аккуратна и документальна, а не потому, что заявляет несуществующие подключения.

## 1. Мировая база стандарта

Стандарт собран на канонических первоисточниках (это стабильные общедоступные документы; ссылки — канонические):

- **GOV.UK Design Principles / Service Manual** — https://www.gov.uk/guidance/government-design-principles — принцип №4: *«Do the hard work to make it simple»* — простота для пользователя достигается сложной работой команды, а не урезанием функций. Государственные сервисы Великобритании — мировой эталон «сложное ведомство → простой экран».
- **W3C, Making Content Usable for People with Cognitive and Learning Disabilities (COGA)** — https://www.w3.org/TR/coga-usable/ — ключевой документ по проектированию «для любого уровня подготовки»: понятная цель каждого экрана, знакомые слова, помощь в месте действия, защита от ошибок, не полагаться на память пользователя.
- **WCAG 2.2** — https://www.w3.org/TR/WCAG22/ — контраст 4.5:1, размеры целей 24px минимум (мы берём 48px для полевых ролей), видимый фокус, отсутствие информации «только цветом».
- **ISO 9241-110 (принципы диалога)** — пригодность для задачи, самоописательность, соответствие ожиданиям, устойчивость к ошибкам.
- **Nielsen Norman Group, 10 эвристик юзабилити** — https://www.nngroup.com/articles/ten-usability-heuristics/ — статус системы всегда виден; язык пользователя, а не системы; предотвращение ошибок; распознавание вместо припоминания.
- **PlainLanguage.gov (федеральный стандарт простого языка США)** — https://www.plainlanguage.gov/guidelines/ — короткие предложения, активный залог, одно понятие = одно слово.
- **Законы восприятия:** Хика (меньше одновременных вариантов — быстрее выбор), Миллера (7±2 объекта в рабочей памяти), Фиттса (крупные близкие цели быстрее).

Примечание: в текущей сессии живой доступ к вебу был ограничен (лимит поиска), источники приведены по каноническим адресам и содержанию.

## 2. Стандарт простоты — 12 обязательных правил

Каждое правило имеет проверяемый критерий приёмки. Экран не принимается, если нарушено хотя бы одно.

**П1. Один экран — одна задача, одно главное действие.**
На экране ровно одна основная кнопка (заливка бренд-цветом). Все прочие действия — второй ранг (контурные) или в меню «Ещё». *Критерий: на скриншоте любой страницы можно за 5 секунд ответить «что здесь надо сделать».*

**П2. Сложность складывается, а не удаляется (progressive disclosure).**
Каждая существующая функция достижима максимум за 2 клика от своего экрана (раскрытие «Подробнее», меню «Ещё», вкладка «Все параметры»). *Критерий: карта функций до/после волны миграции — 100% совпадение; e2e-тесты фиксируют достижимость.*

**П3. Язык шестого класса.**
Фразы ≤ 12 слов, активный залог, обращение на «вы». Запрещены: канцелярит («осуществить», «данный», «функционал»), внутренний жаргон («контур», «кокпит», «рейл»), непереведённые термины (не «RFQ», а «Запрос цены»; профессиональный термин — в скобках при первом употреблении). *Критерий: словарь платформы (§5) — единственный источник терминов; CI-проверка текстов по стоп-словарю.*

**П4. Одно понятие — одно слово везде.**
Сделка, лот, заявка, допуск, приёмка, проба, спор — по одному слову на понятие на всех 225 страницах. *Критерий: линтер словаря по i18n-файлам.*

**П5. Не полагаться на память пользователя (COGA).**
Всё, что нужно для решения, показано на месте решения: остаток по сделке — рядом с кнопкой оплаты, требования к качеству — рядом с результатом пробы. Никаких «см. раздел X».

**П6. Ошибка — это инструкция, а не приговор.**
Формат любой ошибки: **Что случилось → Почему → Что сделать** + кнопка этого действия. Тупиковые экраны запрещены; на каждом состоянии есть выход («Вернуться к сделке», «Позвать оператора»). *Критерий: компонент ошибки один на платформу, свободный текст ошибок запрещён.*

**П7. Необратимое действие подтверждается пересказом.**
Перед подписанием/оплатой/расторжением — простой пересказ последствий («Вы переведёте 1 250 000 ₽ на счёт ООО «Колос». Вернуть деньги без согласия продавца будет нельзя») + явная кнопка. Двойное подтверждение только для денег и расторжений — не для всего подряд.

**П8. Статус всегда виден и назван словом.**
Глобальных статусов сделки ≤ 7. Каждый статус = слово + иконка + цвет (никогда только цвет — WCAG 1.4.1). Статус и следующий шаг видны на каждом экране сделки (хребет из аудита, §4).

**П9. Формы: одна колонка, ≤ 7 полей на шаг, черновик сохраняется сам.**
Длинные формы режутся на пронумерованные шаги («Шаг 2 из 4: Груз»). Потеря введённого — запрещена. Поля объясняются примером («ИНН — 10 цифр, есть в карточке организации»), а не звёздочкой.

**П10. Навигация: ≤ 5 пунктов первого уровня на роль.**
Всё остальное — внутри разделов и через поиск (⌘K / кнопка «Найти»). 110 маршрутов остаются, но пользователь видит пять дверей, а не сто десять.

**П11. Помощь в месте действия.**
У каждого нетривиального элемента — «Что это?» (раскрытие на месте, одним предложением из словаря). Отдельный раздел «Помощь» остаётся, но им нельзя оправдывать непонятный экран.

**П12. Физическая доступность.**
Тач-цели ≥ 48 px (полевые роли), шрифт ≥ 16 px на мобильном, контраст ≥ 4.5:1, видимый фокус, полная работа с клавиатуры, `prefers-reduced-motion` уважается всегда.

## 3. Канон «промышленной федеральной системы»

Взрослость — это документальность и предсказуемость, а не серость. Обязательные признаки:

1. **У всего есть номер.** Сделка № ПЦ-2026-018342, лот, заявка, спор, документ — номер виден в заголовке, копируется одним нажатием, по нему всё ищется. Номер — язык, на котором люди говорят по телефону.
2. **У всего есть время и автор.** Каждое событие: дата, время с минутами и часовым поясом, ФИО/организация. Формат дат единый: «19 июля 2026, 14:32 (МСК)».
3. **Служебная строка системы** (внизу шелла, мелко, всегда): время последнего обновления данных, номер версии системы, «Действия в системе журналируются». Это тихий сигнал промышленности — и он честен (аудит-журнал в коде есть).
4. **Документы выглядят как документы.** Печатные формы, PDF-выгрузка, штамп-«пломба» доказательства (хеш, время, подписант) — единый компонент EvidenceSeal из аудита §5.3.
5. **Таблица — первоклассный жанр.** Моноширинные цифры, выравнивание по разрядам, итоги, экспорт. Никаких «карточек с тенями» там, где оператор работает со списком из 200 строк.
6. **Визуальная тишина.** Запрещены в рабочем контуре: маркетинговые градиенты, свечение, эмодзи, анимация ради анимации, «версии» в глазах пользователя (v7/v9 — только в коде). Палитра — токены аудита §5.1: нейтральные поверхности + семантические доменные цвета.
7. **Предсказуемость.** Один и тот же элемент в одном и том же месте на каждом экране: главное действие — справа снизу формы, статус — в шапке, помощь — у поля. Пользователь учится один раз.
8. **Честная зрелость.** Регламентные формулировки о стадии работы сервисов (`AGENTS.md`) сохраняются, но переводятся с жаргона на язык пользователя: не «Контур проверки», а «Раздел работает в режиме опытной эксплуатации». Это норма федеральных систем и признак взрослости, а не слабости.

## 4. Анти-демо программа

Что сегодня создаёт «демо-ощущение» и что с этим делать (функции сохраняются, меняется адрес и подача):

| Элемент кода | Проблема | Предписание |
|---|---|---|
| `demo-banner.tsx`, `system-mode-banner.tsx`, `runtime-source-banner.tsx` | три баннера кричат «это демо» поверх работы | один тихий индикатор режима в служебной строке (§3.3); полные сведения — на странице «О системе» |
| `feature-flag-strip.tsx` | флаги фич на глазах пользователя | перенести в staff-зону, доступ по роли |
| `simulation-console.tsx`, `map-simulation.tsx` | «симуляция» в рабочем словаре | переименовать в «Тренировочный режим», доступ из staff/обучения, из рабочих кабинетов убрать |
| Бейджи «Рабочий контур / Контур проверки» в шелле | внутренний жаргон стадий | формулировки §3.8; бейдж остаётся, слова меняются |
| `data-testid='…-execution-cockpit'` и подобные названия в UI-текстах | техножаргон просачивается в интерфейс | testid — можно, пользовательские тексты — по словарю |
| Названия версий (platform-v7, v9) в URL и заголовках | пользователь видит строительные леса | заголовки/тексты без версий; URL мигрируют в последнюю очередь с redirect'ами |
| Пустые графики и «0 записей» без объяснения | ощущение муляжа | каждое пустое состояние: что это + какое первое действие (компонент EmptyState уже существует) |

## 5. Словарь платформы (старт, расширяется)

Один термин = одна русская формулировка + одно определение в одну строку («Что это?» берёт текст отсюда):

| Термин | Слово в интерфейсе | Определение для подсказки |
|---|---|---|
| RFQ | Запрос цены | Вы описываете, что хотите купить — продавцы присылают цены |
| Deal / сделка | Сделка | Договор между покупателем и продавцом, который платформа ведёт по шагам |
| Lot | Лот | Партия зерна, выставленная на продажу |
| Escrow / деньги | Защищённый платёж | Деньги заморожены и уйдут продавцу только после приёмки |
| Evidence | Доказательство | Документ или запись, подтверждающие факт: взвешивание, проба, оплата |
| Dispute | Спор | Разногласие по сделке, которое разбирает арбитр |
| Admission | Допуск | Проверка организации перед участием в сделках |
| Survey | Осмотр груза | Независимая проверка количества и качества при отгрузке |

Правило пополнения: новый термин не попадает в интерфейс, пока не внесён сюда.

## 6. Пять канонических шаблонов экрана

Все 225 страниц обязаны сводиться к одному из пяти шаблонов (иное — согласование через governance):

1. **Обзор роли** — «моя смена»: 1 фокус-карточка «ваше следующее действие», 3–4 показателя, список требующего внимания.
2. **Список** — таблица/карточки с поиском, фильтрами вторым рангом, одним действием на строку.
3. **Карточка сделки** — хребет этапов сверху, слева содержание этапа, справа колонка фактов (номера, суммы, сроки, документы).
4. **Мастер (шаги)** — пронумерованные шаги, ≤ 7 полей, автосохранение, пересказ перед подтверждением.
5. **Документ** — печатный вид, пломбы, выгрузка PDF.

## 7. Приёмочные испытания понятности

Волна миграции (аудит §6) считается завершённой, только если:

1. **5-секундный тест:** посторонний человек по скриншоту называет назначение экрана и главное действие. ≥ 8 из 10 ответов верны.
2. **Тест первой задачи:** новый пользователь роли без обучения доводит ключевую задачу до конца ≤ 3 минут, 0 обращений к помощи. (Ключевые задачи: покупатель — создать запрос цены; продавец — выставить лот; водитель — подтвердить погрузку; оператор — разобрать инцидент.)
3. **Тест чтения:** все тексты экрана проходят стоп-словарь (П3) в CI.
4. **Тест полноты:** автоматическое сравнение карты функций до/после — 100%.
5. **Тест доступности:** axe-core 0 критических; контраст и фокус — визуальной проверкой.
6. **Анти-демо тест:** на экранах рабочих ролей нет ни одного элемента из таблицы §4.

## 8. Порядок применения

- Стандарт применяется по волнам плана миграции из `AUDIT_design-visual_2026-07-19.md` §6 — начиная с волны 1 (вход и публичный контур).
- Противоречие между красотой и понятностью решается в пользу понятности; между понятностью и полнотой функций — добавлением ранга (П2), а не удалением.
- Изменения стандарта — только через PR с пометкой владельца продукта.
