# PC-CROP-01A — Commodity Profile domain and interface foundation

Статус: `IMPLEMENTED_FOR_EXACT_HEAD_ACCEPTANCE`  
Issue: `#2875`

Operational status: `NOT_ATTESTED`

## 1. Назначение slice

Создать обязательный доменный контракт и интерфейсную систему товарных профилей до изменения PostgreSQL schema и открытия runtime API. Это не mock-реестр и не параллельный источник истины. Компонент принимает только данные и разрешённые действия, рассчитанные сервером; подключение PostgreSQL-authority выполняется следующим совместимым slice.

Целевая система проектируется для федеральной промышленной эксплуатации. Термин `NOT_ATTESTED` означает только отсутствие полного exact-release эксплуатационного evidence, а не снижение требований до демонстрационного или пилотного уровня.

## 2. Доменное решение

### 2.1. Шесть архетипов

- `DRY_BULK` — зерно, масличные, зернобобовые и сыпучие кормовые.
- `SEED_PLANTING` — семена и посадочный материал.
- `ROOT_INDUSTRIAL` — картофель, корнеплоды и технические культуры.
- `FRESH_PACKED` — овощи, фрукты, ягоды и зелень в таре/палетах.
- `GREENHOUSE_RECURRING` — повторяющиеся поставки защищённого грунта.
- `ORGANIC_EXPORT_QUARANTINE` — органические, экспортные и карантинно-контролируемые партии.

Архетип задаёт конфигурационную рамку, но не создаёт отдельную state machine или отдельный `Deal`.

### 2.2. Версия профиля

`CommodityProfileVersion` содержит:

- стабильный `profileId` и уникальный `versionId`;
- последовательность версии;
- lifecycle state;
- effective range;
- точный SHA-256 содержания;
- коды продукции;
- единицы и конверсии;
- показатели качества и методы;
- правила отбора и хранения проб;
- документы, подписи, реестры и release blockers;
- условия хранения, упаковки, срока годности и смешения;
- правила приёмки и быстрого спора;
- source references и legal rule IDs.

### 2.3. Инварианты

1. JavaScript `number` не является authority для нормативных и количественных конверсий. Используются exact decimal strings с максимум шестью знаками.
2. Для каждой представленной размерности существует ровно одна базовая единица.
3. Нижняя граница не может превышать верхнюю.
4. Release blocker обязан ссылаться на объявленное документное требование.
5. Опубликованная версия неизменяема; исправление создаёт новую версию.
6. `EFFECTIVE` требует effective date и evidence согласования.
7. Две approved/effective версии не могут пересекаться по времени.
8. Содержание нормализуется перед hash, поэтому порядок set-подобных правил не меняет SHA-256.
9. Возвращаемый объект глубоко заморожен, включая вложенные списки и правила.
10. Ошибки имеют стабильный `code` и `path` для API, аудита и RU/EN/ZH интерфейса.

## 3. Интерфейсная архитектура

### 3.1. Главная задача пользователя

Пользователь должен за несколько секунд ответить:

- какой профиль действует;
- для каких культур и сценария исполнения;
- какая версия закрепляется за Сделкой;
- какие показатели качества обязательны;
- какие документы и реестры блокируют приёмку или выпуск денег;
- какие сроки, температура, упаковка и правила смешения применяются;
- насколько актуальны источники;
- какое действие разрешено пользователю сервером сейчас.

### 3.2. Desktop

```
┌──────────────────────────────────────────────────────────────────────┐
│ Заголовок + граница достоверности       KPI: всего / действует / risk│
├──────────────────────────────────────────────────────────────────────┤
│ Поиск                 Состояние       Архетип       Создать черновик │
├───────────────────────────────┬──────────────────────────────────────┤
│ Компактный реестр             │ Выбранный профиль                   │
│ Профиль | state | v | source  │ state · source · immutable/pinned   │
│                               │ разрешённое сервером действие        │
│                               │ качество / документы / хранение      │
│                               │ приёмка / provenance / history       │
└───────────────────────────────┴──────────────────────────────────────┘
```

Реестр остаётся таблицей, а не стеной крупных карточек. Детали раскрываются progressively, чтобы сложность правил не конкурировала с главным решением.

### 3.3. Mobile 320–430 px

Порядок:

1. Заголовок и честная граница источников.
2. Компактные KPI строками.
3. Поиск.
4. Фильтр состояния.
5. Фильтр архетипа.
6. Реестр как доступные stacked rows.
7. Выбранный профиль.
8. Одно разрешённое действие.
9. Сворачиваемые группы правил.

Кнопки и поля имеют минимальную высоту 44 px. Горизонтальная прокрутка таблицы на узком экране не требуется.

## 4. Состояния интерфейса

| Состояние | Поведение |
|---|---|
| `loading` | Сообщает, что загружаются версии, источники и серверные права, а не рисует skeleton как готовые факты. |
| `empty` | Объясняет, что Сделка не должна стартовать без профиля; показывает создание только при серверном полномочии. |
| `error` | Не заменяет отсутствие данных ложным «всё в порядке»; предлагает повторную загрузку. |
| `forbidden` | Показывает отсутствие полномочия без раскрытия закрытых данных. |
| `conflict` | Сообщает об optimistic-concurrency конфликте и требует получить актуальную серверную версию. |
| `ready` | Показывает compact registry и выбранный профиль. |
| filter empty | Сохраняет загруженный экран и сообщает, что нет совпадений по фильтрам. |

## 5. Lifecycle action UX

Интерфейс не вычисляет право изменения. Сервер передаёт один `primaryAction`:

- code;
- локализованную метку;
- reason/disabledReason;
- owner;
- deadline;
- impact;
- признак обязательного confirmation;
- disabled state.

Для consequential actions дальнейший runtime обязан реализовать:

`review inputs → показать diff и последствия → проверить prerequisites → явное подтверждение → server command → receipt/audit`.

Компонент не меняет state локально и не выбирает роль, tenant или approval authority.

## 6. Визуальная система

- Основа — Design System v8 и текущие токены платформы.
- Главный акцент — зелёный платформы; красный только для revoked/blocking/critical.
- Никаких декоративных отраслевых иллюстраций, градиентных «AI-карточек» или тяжёлой панели администратора.
- Типографика строит иерархию: решение → состояние → версия → правило → provenance.
- Hash, codes и version IDs отображаются моноширинно, но не доминируют.
- Опубликованная/закреплённая версия имеет явный immutable marker.
- Состояние источника визуально отдельно от lifecycle state, чтобы «профиль действует» не трактовалось как «внешний источник подтверждён live».

## 7. Доступность

- семантическая таблица на desktop;
- реальные `button`, `input`, `select`, `details/summary`;
- `aria-current` для выбранного профиля;
- `aria-live` для смены выбранных данных;
- `role=alert` для error/conflict;
- keyboard focus ring;
- forced-colors;
- reduced-motion;
- touch targets 44 px;
- RU/EN/ZH без смешения языка внутри одной локали.

## 8. Безопасность и честность

- Нет `fetch`, static fixtures, localStorage или sessionStorage внутри presentational component.
- Нет client-selected role/tenant.
- Нет автоматической публикации или активации.
- Нет заявления о production-доступности ФГИС, ЭДО, банка или реестра.
- Нет прямого write из TAI.
- Неverified source получает `REVERIFY_REQUIRED` или `BLOCKED_EXTERNAL`, а не зелёный статус.
