# Прогон пилотной сделки DL-9106 по всем кабинетам

Один сквозной сценарий: от захода человека на платформу до закрытия сделки и
выпуска денег. Для каждого шага указано: **роль/кабинет**, **что происходит**,
**что сейчас симулируется** и **шов под реального партнёра** (где разработчик
подменит мок на боевой контур, см. `developer-handoff-real-partner-integration.md`).

Канонический кейс — единая «точка правды» `deal-execution-source-of-truth.ts`:
- Сделка **DL-9106**, лот **LOT-2403**, пшеница 4 класса, **600 т**, 16 080 ₽/т.
- Резерв **9 648 000 ₽**, к выпуску **0 ₽** до закрытия оснований.
- Рейсы **TRIP-2403-001..003**, заявка **LOG-REQ-2403**.

Согласованность данных между всеми кабинетами проверяется тестом
`tests/unit/dl9106CrossRoleConsistency.test.ts`.

## 0. Вход на платформу

- **Экран входа** `/platform-v7` → витрина «Одна сделка. Полный контроль.» +
  «Выберите свою роль» по группам. Регистрация `/register`, вход `/login`.
- Симуляция: вход открыт во все кабинеты (без боевого гейтинга доступа).
- Шов под реального партнёра: авторизация организаций (СберБизнес ID / ЕСИА) —
  `providerRegistry.ts` (`sber_business_id`), статус `requires_access`.

## 1. Продавец — `/platform-v7/seller`

- Публикует лот LOT-2403, ждёт встречного предложения; оформляет СДИЗ, готовит
  документы, отслеживает основание для денег.
- Симуляция: статусы СДИЗ во ФГИС (`fgis.*` в каноническом кейсе), черновики
  документов.
- Шов: **ФГИС «Зерно»** (`fgis`) — `external-adapters.ts`/`fgis-adapter-emulator.ts`;
  **ЭДО** (`edo`) для договора/УПД. До подключения — `requires_connection`.

## 2. Покупатель — `/platform-v7/buyer`

- Запрос → оффер → резерв → приёмка/качество → погашение СДИЗ → подписание УПД.
- Симуляция: статус «готов к резерву», карточка денег (`/deals/DL-9106/money`).
- Шов: **Банк** (`bank`) для резерва; **ФГИС** для погашения СДИЗ покупателем.

## 3. Логистика — `/platform-v7/logistics`

- Назначает рейс, перевозчика, окно погрузки; собирает транспортный пакет ЭТрН.
- Симуляция: 3 рейса и пакет ЭТрН из канонического кейса (`logistics.trips`,
  `logistics.epdPackage`).
- Шов: **ГИС ЭПД / ЭТрН** (`epd`) и **GPS-телематика** (`logistics`) —
  `external-adapters.ts`; мок-фабрики, боевой адаптер по образцу.

## 4. Водитель — `/platform-v7/driver/field`

- Полевой режим: один текущий рейс и одно следующее действие (фото пломбы, статус).
- Симуляция: `runtime/driver-cockpit-state.ts` поверх канонического рейса.
- Шов: **GPS** (`logistics`) для событий маршрута; подпись водителя в **ЭПД**.

## 5. Элеватор — `/platform-v7/elevator`

- Приёмка: фиксирует вес, отклонение, акт расхождения, передаёт пробу в лабораторию.
- Симуляция: приёмочная сцена (вес 600→598,8 т, отклонение −1,2 т, показатели).
- Шов: **весовой/складской контур элеватора** + влияние веса на деньги —
  доменная функция `calculateElevatorWeightImpact()` (остаётся при боевом адаптере).

## 6. Лаборатория — `/platform-v7/lab`

- Проба → протокол качества → отклонение и влияние на расчёт.
- Симуляция: протокол «ожидается», показатели качества.
- Шов: **лаборатория** (`lab`) — `external-adapters.ts`; влияние качества —
  `calculateLabQualityImpact()`.

## 7. Сюрвейер — `/platform-v7/surveyor`

- Осмотр, фото, расхождения, заключение как доказательство.
- Симуляция: сцена осмотра/доказательств.
- Шов: загрузка доказательств может идти через хранилище/ЭДО партнёра.

## 8. Банк — `/platform-v7/bank`

- Резерв → удержание → ручная проверка основания → выпуск/возврат.
- Симуляция: «резерв ожидает подтверждения»; деньги не выпускаются без полного
  основания (СДИЗ, ЭТрН, ГИС ЭПД, приёмка, качество).
- Шов: **Банк** (`bank`, эскроу/безопасная сделка) — главный денежный адаптер;
  подтверждение приходит ответным событием + сверка (`money-runtime`).

## 9. Оператор — `/platform-v7/operator`

- Очередь блокеров, маршрутизация к ответственной роли, следующее действие.
- Симуляция: рантайм-очередь блокеров из канонического состояния.
- Шов: интеграционные события партнёров наполняют очередь (через адаптеры).

## 10. Руководитель — `/platform-v7/executive`

- Деньги под риском, узкие места, зрелость пилота (BI-пульт).
- Симуляция: `runtime/bi-cockpit-state.ts` (GMV, метрики runtime/scenario).
- Шов: боевые показатели появятся при подключении реального рантайма/БД.

## 11. Арбитр — `/platform-v7/arbitrator`

- Спор → доказательства → решение по деньгам (удержание/возврат/выпуск).
- Симуляция: сцена спора и решения; влияние на деньги.
- Шов: решение применяется к банковскому основанию через
  `applyArbitrationOutcomeToBankBasis` (тот же банковский адаптер).

## 12. Комплаенс — `/platform-v7/compliance`

- KYC/AML, риск, стоп-факторы, блокер сделки.
- Симуляция: сцена проверки контрагента.
- Шов: **проверка контрагента** (`counterparty_check`, `manual_review`) —
  внешний источник/реестр.

## Закрытие сделки

Деньги выпускаются только после закрытия всех оснований (документы, СДИЗ, ЭТрН,
ГИС ЭПД, приёмка, качество) и подтверждённого банковского события. До этого
`readyToReleaseAmount = 0`, `releasedAmount = 0`. Это фиксирует
`selectDealMoneyState('DL-9106')` и проверяет кросс-ролевой тест.

## Что остаётся для реального пилота (§43, owner-side)

Подключение фабрики адаптеров и швы готовы (см. developer-handoff). Вне кода:
договоры, API-доступы и credentials, номинальный счёт, юрпроверка, security
review и реальные пилотные сделки. После этого — реализация боевых адаптеров по
образцу `adapters/real-adapter-template.ts` и регистрация боевого реестра.
