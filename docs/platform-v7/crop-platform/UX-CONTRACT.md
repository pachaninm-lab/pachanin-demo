# UX Contract: растениеводство внутри единого исполнения Сделки

## 1. Главный принцип

После входа пользователь видит не каталог модулей, а очередь намерений:

1. Что происходит сейчас.
2. Что требует моего действия.
3. Что заблокировано и почему.
4. Сколько товара или денег под риском.
5. Кто владелец следующего шага и какой срок.
6. Какое действие безопасно выполнить сейчас.

Навигация остаётся доступной, но не конкурирует с операционной задачей.

## 2. Информационная архитектура

### 2.1. Home = Priority Queue

Первый экран содержит четыре слоя в фиксированном порядке:

- **Критично сейчас** — просроченные действия, blocking disputes, документные и денежные blockers.
- **Сегодня** — прибытия, слоты, отбор проб, подписи, release readiness.
- **Сделки в работе** — компактные строки с этапом, владельцем, сроком и суммой/массой риска.
- **Обзор** — KPI и тренды только после операционной очереди.

Запрещено превращать Home в сетку крупных равноправных карточек. Один объект = одна строка решения; детали раскрываются по требованию.

### 2.2. Deal Workspace

На desktop рабочая область состоит из пяти зон:

1. **Header strip** — Deal ID, продукт, стороны, pinned policy, общий health.
2. **Current state** — параллельные dimensions: legal, money, physical, quality, documents, dispute.
3. **Next action rail** — одно главное действие, owner, deadline, blockers, последствия.
4. **Execution timeline** — события, evidence, внешние статусы и corrective versions.
5. **Context drawer** — документы, партия/genealogy, качество, деньги, TAI, аудит.

На mobile зоны становятся последовательностью: `Next action → blockers/risk → current state → timeline → context`.

### 2.3. Commodity Profile UI

Пользователь не редактирует технический JSON. Экран профиля использует progressive disclosure:

- идентичность продукта и назначение;
- единицы и конверсии;
- показатели качества и методы;
- документы/реестры;
- хранение/температура/shelf-life/тара;
- правила приёмки и price delta;
- release blockers;
- история версий и approval trail.

Каждое поле показывает источник, effective date, обязательность, уровень доверия и влияние на Сделку. Published/pinned version нельзя тихо изменить.

## 3. Универсальный паттерн критического действия

Каждое consequential действие проходит один и тот же контракт:

`Review → impact → prerequisites → confirmation → server result → immutable receipt`

Обязательные состояния:

- `READY` — действие разрешено;
- `BLOCKED` — причина + конкретный способ снять blocker;
- `PENDING_EXTERNAL` — команда принята, внешний результат ожидается;
- `PARTIAL` — часть эффекта подтверждена, остаток виден;
- `CONFLICT` — версия устарела, показан актуальный snapshot;
- `DENIED` — отказ policy с безопасным reason code;
- `FAILED_RETRYABLE` — повтор возможен без дублирования эффекта;
- `FAILED_FINAL` — нужен оператор/другой путь;
- `SUCCEEDED` — receipt, correlationId и связанное событие.

Disabled button без объяснения запрещён.

## 4. Ролевой mobile-first минимум

| Роль | Первый экран | Главное действие | Нельзя показывать |
|---|---|---|---|
| Производитель/продавец | готовность партии, документы, рейсы, деньги | снять ближайший blocker | закрытые ставки, банковую внутреннюю логику |
| Покупатель | прибытия, качество, документы, release readiness | принять/оспорить подтверждённый факт | чужие ставки и закрытые агроданные |
| Логист | очередь заявок, слоты, ETA, инциденты | назначить/восстановить рейс | цену, кредит, лишние данные сторон |
| Водитель | один активный рейс | выполнить следующий checkpoint | деньги, аукцион, чужие сделки |
| Приёмщик/элеватор | скан рейса, вес, проба, очередь | зафиксировать физический факт | банковые и коммерческие детали |
| Лаборант | очередь проб | внести метод/результат/протокол | коммерческие условия вне необходимости |
| Банк | obligations, reserve, blockers, reconciliation | подтвердить банковский event | кнопки выпуска без закрытых условий |
| Оператор | cross-deal blocker queue | управляемое устранение причины | прямое редактирование authority |
| Арбитр/сюрвейер | назначенные дела/evidence | структурированное заключение | данные вне предмета спора |
| Руководитель | деньги/товар под риском, SLA, тренды | drill-down к владельцу | технический шум адаптеров |

## 5. Offline contract

Полевой интерфейс всегда различает:

- `LOCAL_ONLY` — сохранено только на устройстве;
- `SYNC_QUEUED` — ожидает передачи;
- `SERVER_ACCEPTED` — сервер принял команду;
- `CANONICAL_CONFIRMED` — событие вошло в authority timeline;
- `SYNC_CONFLICT` — требуется сверка;
- `REVOKED_DEVICE` — локальные privileged действия запрещены.

Очередь шифруется, имеет device identity, commandId, idempotency key, source time и media hash. UI не показывает локальную запись как подтверждённый факт.

## 6. External systems

Пользователь видит бизнес-состояние, а не SOAP/HTTP детали:

- `Работает` — confirmed-live и fresh health;
- `Ожидаем ответ` — durable pending;
- `Восстанавливаем обмен` — backlog/retry;
- `Нужна проверка` — quarantine/reconciliation mismatch;
- `Недоступно в этом контуре` — нет договора/access/provider.

Техническая диагностика доступна оператору в context drawer. Ошибка внешней системы не формулируется как ошибка пользователя.

## 7. TAI в интерфейсе

TAI не является отдельным лабиринтом экранов. Он встроен в контекст:

- объясняет статус и blocker;
- сравнивает подтверждённые варианты;
- показывает источник и свежесть;
- готовит draft/prepared action;
- связывает рекомендацию с конкретной Deal dimension;
- отказывается при недостатке данных или запрете policy.

TAI не скрывает primary action, не меняет аргументы после confirmation binding и не создаёт визуальную иллюзию уже выполненного действия.

## 8. Visual system

- Один основной акцент действия; цвет не является единственным носителем смысла.
- Статусы используют icon + label + reason; не более трёх уровней визуальной иерархии на экране.
- Компактные data rows предпочтительнее крупных карточек.
- Числа выравниваются таблично; единица всегда рядом; исходное и нормализованное значение различимы.
- Таймлайн отделяет факт, внешнее подтверждение, рекомендацию и гипотезу.
- Motion только для причинной связи; `prefers-reduced-motion` обязателен.
- Mobile targets не меньше 44×44 CSS px; safe-area учитывается.

## 9. Accessibility и локализация

- WCAG 2.2 AA: keyboard, focus, screen reader, contrast, error association, reduced motion.
- Полная работоспособность 320/375/390/430 px и desktop.
- RU — юридически авторитетный текст; EN/ZH — локализованный интерфейс и пояснение.
- Код, единица, нормативный идентификатор и подписываемое значение не переводятся произвольно.
- Длинные RU/EN/ZH строки тестируются без обрезания критического смысла.

## 10. UI acceptance per slice

Новый экран не считается готовым без:

- loading / empty / error / permission / blocked / offline / conflict / success states;
- mobile screenshots 320, 375, 390, 430 px;
- keyboard and screen-reader path;
- RU/EN/ZH coverage;
- server-authoritative action test;
- tenant/role negative tests;
- analytics event dictionary без PII/secret leakage;
- measured task completion baseline.
