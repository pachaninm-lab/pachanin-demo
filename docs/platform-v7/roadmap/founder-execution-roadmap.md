# Founder Execution Roadmap — от пилота к боевой сделке

Назначение: единый план «что делать дальше» от лица основателя-разработчика.
Состояние на момент написания: интерфейс, UX и внутренний контур исполнения
готовы и отполированы; деньги и внешние подтверждения — симуляция. Следующий
рубеж не дизайн, а перевод одного контура из симуляции в боевой.

Для каждого пункта: **что это**, **код-готовность** (что уже есть в репозитории)
и **owner-side** (что вне кода — доступы, договоры, инфраструктура, юр.).

---

## Тир 1 — до первой реальной сделки (критический путь)

### 1. Боевой банк / эскроу (ядро ценности)
- **Код-готов:** шов `lib/platform-v7/adapter-factory.ts`
  (`platformV7CreateAdapterRegistry` выбирает mock/real по окружению),
  образец `lib/platform-v7/adapters/real-adapter-template.ts`,
  контракт `external-adapters.ts` (`PlatformV7BankRequest`), гейты честности
  `integrations/providerRegistry.ts` (`canClaimProviderLive`). Тест шва зелёный.
- **Сделать (код):** реализовать `createRealBankAdapter` (реальные HTTP-вызовы),
  зарегистрировать боевой реестр `platformV7RegisterRealAdapterRegistry(...)` в
  точке старта приложения, выставить `NEXT_PUBLIC_PLATFORM_V7_ENVIRONMENT=production`.
- **Owner-side:** договор с банком, номинальный счёт, credentials, callback-URL
  для подтверждений, банковская песочница для первого escrow-цикла.

### 2. Реальная аутентификация + гейтинг доступа
- **Код-готов:** ролевые кабинеты разделены; модель доступа `access-control.ts`
  (`PlatformV7AccessActor`/роли/ресурсные скоупы) уже используется рантаймом.
- **Сделать (код):** включить реальную проверку identity (СберБизнес ID / ЕСИА),
  закрыть «вход открыт во все ЛК», применять RBAC: организация видит только свои
  сделки/документы/деньги.
- **Owner-side:** доступы к провайдеру идентификации, согласование скоупов.

### 3. Реальная персистентность (БД)
- **Код-готов:** порты `runtime/persistence-ports.ts` (`P7RuntimeUnitOfWork`,
  8 репозиториев, идемпотентность/версии/аудит); мок `mock-persistence-adapter.ts`.
- **Сделать (код):** реализовать адаптер БД с теми же сигнатурами портов;
  прикладной сервис (`runtime/application-service.ts`) не меняется.
- **Owner-side:** провижининг БД, бэкапы, миграции, мониторинг.

### 4. Юридический и security-блок (параллельно, бизнес)
- Договоры ФГИС «Зерно» / оператор ЭДО / ГИС ЭПД / банк; 152-ФЗ (ПДн),
  115-ФЗ; security review денежного и доказательного контура.

## Тир 3 — зрелость и масштаб

8. **Консолидировать два хедера** (`CockpitHero` ↔ `ExecutionDesignSystem`) в один
   дизайн-стандарт — меньше поддержки, гарантированно ровная подача.
9. **Реальные e2e в CI** (Playwright по ключевым ролевым маршрутам на каждом PR;
   сейчас не required-гейт).
10. **GTM/продукт:** take-rate/комиссия (в BI пока сценарная,
    `runtime/bi-cockpit-state.ts`), онбординг организаций, поддержка/SLA, тарифы.

## Что готово к «переключению флага» уже сейчас
- Шов адаптеров mock↔real + образец + handoff-карта
  (`docs/platform-v7/integrations/developer-handoff-real-partner-integration.md`).
- Порты персистентности с идемпотентностью/версиями/аудитом.
- Гейты честности: боевой статус провайдера/денег нельзя показать без договора,
  credentials, callback-ов и подтверждённых операций.
- Полный UX/визуал: единый премиум-герой, микро-анимации, фокусные мобильные
  экраны, WCAG 2 AA, навигация без тупиков, чистая гидратация.

## Первый коммит «как основатель»
Реализовать боевой банковский адаптер за фиче-флагом (`production` → real registry)
и провести один escrow-цикл в банковской песочнице. Это первый камень, который
превращает «симуляцию» в «продукт»; всё остальное под него уже подготовлено.
