# Netlify post-deploy checklist for platform-v7

Цель: после каждого Netlify deploy проверять не только зелёный статус сборки, но и фактическую пригодность `/platform-v7` для controlled pilot.

## 1. Commit

- В Netlify production deploy указан свежий commit из `main`.
- Commit совпадает с последним ожидаемым GitHub merge commit.
- Старые failed deploy не используются как источник статуса.

## 2. Main route

Проверить:

- `https://vermillion-kitsune-0e7b97.netlify.app/platform-v7/`
- страница открывается без 404/500;
- первый экран публичный, а не кабинетный;
- есть логотип проекта;
- нет bottom nav на публичном первом экране;
- нет внутренних кабинетных CTA до входа в контур.

## 3. Open route

Проверить:

- `https://vermillion-kitsune-0e7b97.netlify.app/platform-v7/open/`
- нет старых CTA: `Выставить партию`, `Создать запрос на закупку`, `Открытый просмотр`;
- есть безопасные действия: выбор роли, вход, демонстрация;
- зрелость описана как пилотный / предынтеграционный контур.

## 4. Mobile 390 x 844

Проверить на телефоне:

- нет горизонтального скролла;
- шапка не перекрывает hero;
- CTA не перекрыт нижней панелью браузера;
- текст читается без приближения;
- карточки не выглядят как мусорная длинная лента.

## 5. Cabinet boundary

Проверить прямой переход без предварительного входа через главный экран:

- `/platform-v7/bank/clean`
- `/platform-v7/deals`
- `/platform-v7/logistics`
- `/platform-v7/driver/field`

Ожидаемая логика: рабочий маршрут не должен открываться как публичная входная точка.

## 6. Copy safety

На публичных экранах не должно быть обещаний, что внешние подключения уже активны или что платформа сама гарантирует расчёт. Корректная рамка: controlled pilot / pre-integration.

## 7. Known limitations

Не считать закрытыми без отдельного подтверждения:

- server-side auth/session для всех runtime операций;
- backend persistence;
- внешние доступы и договоры;
- live credentials;
- подтверждение на реальных сделках;
- внешний security/legal audit.
