# Путь к продакшену: от витрины к работающей платформе

**Дата:** 03.07.2026 · **Основание:** аудит прод-сборки platform-v7 + сверка с кодом репозитория.

## Цель

Превратить текущее состояние («интерактивная витрина»: данные из фикстур в JS-бандле, действия
живут в localStorage, бэкенд написан, но не задеплоен) в работающую платформу: тысячи
пользователей и компаний работают с **общими** сделками через API и одну базу данных.

Калибровка нагрузки: «тысячи одновременных пользователей» ≈ 200–400 запросов/сек.
Это не highload — узкое место не производительность, а **корректность конкурентного
доступа к деньгам** и отсутствие деплоя серверного слоя.

## Ключевой факт: строить почти нечего — нужно задеплоить и сшить

Уже написано в репозитории и используется планом как актив:

| Компонент | Где лежит |
|---|---|
| API (NestJS, 40+ доменных модулей: deals, ledger, disputes, settlement-engine, kyc, storage, auth, mfa…) | `apps/api/src/modules` |
| Схема БД: 28 моделей Prisma (Deal, Shipment, Payment, LedgerEntry, Organization, UserOrg, `tenantId`) + миграции + сиды | `apps/api/prisma` |
| RLS-политики Postgres | `infra/sql/production-rls-policies.sql` |
| Идемпотентность мутаций | `modules/saga/deal-saga.service.ts`, `modules/runtime-core/runtime-money-engine.ts` |
| Transactional outbox | `modules/deals/deal-auto.service.ts` + фронтовые роуты `runtime-outbox*` |
| Наблюдаемость: Sentry, OTel-трейсинг, prom-client, health-контроллер | `apps/api/src/{sentry,tracing}.ts`, `health.controller.ts` |
| Прокси-слой фронта к API | `apps/web/app/api/runtime-*` (ждут `NEXT_PUBLIC_API_URL`) |
| Серверный RBAC по верифицированному JWT (report-only) | `middleware.ts` + `lib/platform-v7/server-cabinet-access` |
| Нагрузочные сценарии | `infra/k6` |

Чего в коде **нет** (проверено): CHECK-констрейнтов на финансовые инварианты,
`SELECT ... FOR UPDATE` в переходах статусов, поля `version` на Deal,
хелпера склонения числительных, миграций persisted-сторов zustand.

## План работ

### Блок А — Деплой и сшивка (критический путь, ~2–3 недели)

1. **РФ-облако для рабочего контура** (152-ФЗ: KYC/AML-данные россиян нельзя держать на
   Netlify/US). Yandex Cloud / VK Cloud / Selectel: managed PostgreSQL + PgBouncer, Redis,
   S3-совместимое хранилище, 2 ВМ под API (2 vCPU / 4 GB) за балансировщиком.
   Лендинг/публичный контур может остаться на Netlify.
2. **Задеплоить `apps/api`** (Dockerfile уже есть в `infra/docker`), прогнать
   `prisma migrate deploy` + сиды. Из docker-compose брать только ядро:
   Postgres, Redis, S3. **Не деплоить** Kafka, ClickHouse, Airflow, Elasticsearch,
   ML-сервис, Istio, Kubernetes — отложить до появления реальной потребности.
3. **Сшить фронт с API:** задать `NEXT_PUBLIC_API_URL` (сейчас дефолт `localhost:4000` —
   в никуда) и общий `JWT_SECRET` для API и web-middleware.
4. **Убрать молчаливый фолбэк.** Прокси-роуты `runtime-*` при недоступности API возвращают
   `{ok:false}` со статусом 200, и UI тихо подставляет фикстуры. Ошибка API → честный
   502/503 и видимое состояние «данные недоступны»; демо-режим — явный флаг с плашкой
   «демонстрационные данные», а не невидимая подмена.
5. **Настоящие сессии и RBAC:** httpOnly-куки, сессии в Redis; серверный RBAC перевести из
   report-only в enforce; декоративную cookie `pc-role` удалить.
6. **Фикстуры → сиды БД** с флагом `is_demo`: контент сохраняется, но становится общим
   и редактируемым. Бизнес-данные убрать из zustand `persist` (localStorage — только
   UI-предпочтения, с `version`/`migrate`).

### Блок Б — Корректность денег (~2 недели, до первого реального рубля)

7. Переходы статусов сделки — только на сервере, в транзакции, с `SELECT ... FOR UPDATE`
   по строке сделки; поле `version` на Deal (оптимистическая блокировка — она же решает
   конфликты офлайн-синка водителя).
8. Финансовые инварианты (`reserved ≥ hold ≥ 0`, баланс леджера) — CHECK-констрейнтами
   в БД, не только в коде.
9. `Idempotency-Key` на всех мутациях (механика в saga/money-engine есть — довести до
   всех эндпоинтов): двойной тап водителя ≠ двойное списание.
10. Конкурентные тесты на существующие saga/outbox/money-engine: гонка двух операторов,
    повторная доставка, падение между шагами.
11. Всё вычислимое (`slaDaysLeft`, просрочки, риск-скоры) — считать на сервере от `now()`,
    не хранить константами.

### Блок В — Онбординг компаний (то, что «объединяет», ~2–3 недели, параллельно Б)

12. Регистрация организации: ИНН → проверка (данные ЕГРЮЛ), создание `Organization`,
    первый пользователь — администратор компании.
13. Приглашения сотрудников по ролям (модели `User`/`UserOrg` уже есть) — водитель,
    лаборант, бухгалтер входят по ссылке-приглашению, а не через демо-переключатель ролей.
14. KYC-поток за существующим модулем `kyc`: до прохождения — витрина и каталог,
    после — сделки и деньги.
15. Изоляция тенантов: включить RLS-политики, каждый видит только свои сделки;
    e2e-тест на невозможность чтения чужой сделки.

### Блок Г — Гигиена и защита (~1 неделя, параллельно)

16. Вернуть проверки сборки: убрать `typescript.ignoreBuildErrors` и
    `eslint.ignoreDuringBuilds` из `apps/web/next.config.js`, разгрести долг типов,
    поставить CI-гейты. Staging-окружение: деплой на каждый merge, прод — по тегу.
17. Быстрые фиксы аудита: `immutable`-кэш на `/_next/static/*`; исключить
    robots/sitemap/статику из middleware; вырезать `console.*` на билде; открыть health;
    один источник security-заголовков (сейчас middleware и next.config противоречат
    друг другу); CSP без `unsafe-eval`.
18. UI-долг из аудита: горизонтальное переполнение (`100vw`→`100%`), хелпер склонения
    числительных, русский глоссарий, дизайн-токены и постепенный вывод `!important`.

### Блок Д — Проверка и запуск (~1 неделя)

19. k6 по трём профилям: каталог лотов, полный цикл сделки, офлайн-синк водителя.
    Цель p95 < 500 мс на 400 RPS. Узкие места ожидаются в N+1 и индексах, не в железе.
20. Включить уже вшитые Sentry/метрики/трейсинг; алерты на ошибки, латентность,
    насыщение пула соединений.
21. Контролируемый пилот: 3–5 реальных компаний, реальные сделки с ручными
    интеграциями, разбор инцидентов → открытие регистрации.

### Отложено сознательно

- **Интеграции** (эскроу-банк, ФГИС «Зерно», ЭДО) — за существующими портами-адаптерами,
  после запуска ядра: требуют договоров и сертификации (месяцы календарного времени),
  но не блокируют запуск с ручными процессами.
- Kafka/ClickHouse/аналитика/ML, Kubernetes, микросервисы, второй регион.
  Второй API-инстанс и read-реплика — только по результатам нагрузочных тестов.

## Definition of Done

- Данные приходят из PostgreSQL через API; в клиентском бандле нет каталога сделок.
- Два пользователя из разных компаний видят одну и ту же сделку и не видят чужие (RLS).
- Действие пользователя переживает закрытие вкладки и видно контрагенту.
- Демо-режим — явный флаг с плашкой; отказ API виден, а не маскируется.
- Конкурентные мутации безопасны: FOR UPDATE + version + идемпотентность + CHECK-инварианты,
  подтверждено тестами.
- Компания проходит путь «регистрация → KYC → сотрудники → первая сделка» без участия
  разработчиков.
- p95 < 500 мс на целевой нагрузке; ошибки в Sentry; health публичный; сборка проходит
  типизацию и линт.
- Рабочий контур и ПДн — в РФ (152-ФЗ).

**Суммарная оценка:** 6–9 недель инженерной работы до пилота с реальными компаниями;
критический путь — блоки А и Б.
