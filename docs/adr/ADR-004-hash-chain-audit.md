# ADR-004: Hash Chain для append-only Audit Trail

**Статус:** Принято  
**Дата:** 2026-02-10  
**Авторы:** Backend Team  

## Контекст

Финансовая платформа требует неизменяемого audit trail для регуляторных требований (152-ФЗ, 63-ФЗ, Росфинмониторинг). Нужно доказать, что история событий не была изменена.

## Решение

Каждая запись аудита содержит **SHA-256 хэш текущей записи + хэш предыдущей**:

```typescript
// SHA-256(id + dealId + eventType + JSON(payload) + prevHash)
hash = createHash('sha256')
  .update(JSON.stringify({ id, dealId, eventType, payload, prevHash }))
  .digest('hex');
```

Применяется в двух местах:
1. **`DealEvent`** — события жизненного цикла сделки
2. **`AuditEvent`** — глобальный audit trail всей платформы

Проверка целостности — при экспорте через `GET /api/exports/deal-report/:dealId` → секция `chainIntegrity`.

## Последствия

- `+` Любая попытка задним числом изменить запись нарушит хэш-цепочку
- `+` Экспортируемые отчёты содержат доказательство целостности
- `+` Минимальный overhead (SHA-256 очень быстрый)
- `-` Хэш рассчитывается в коде, не в БД — требует доверия к коду
- `-` При коррупции early записи все последующие хэши невалидны

## Дополнительные меры

- Ежедневный S3 snapshot audit log с S3 Object Lock (COMPLIANCE mode, 7 лет)
- В будущем: notarization в blockchain (Этап 4)
