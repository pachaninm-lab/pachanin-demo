# Фаза 1 — журнал находок и долгов

Обновляется по ходу работ. Приоритет: B0 — блокирует канонический сценарий,
B1 — искажает продуктовый путь, B2 — техдолг.

## B0 — блокирует сценарий

1. **Tenant-изоляция запрещает торговлю между организациями.**
   Регистрация (`auth.service.register`) создаёт каждой организации собственный
   tenant, а аукционная authority живёт под RLS текущего tenant: покупатель и
   оператор из других tenant'ов получают `AUCTION_LOT_NOT_FOUND` по живому лоту.
   Само-зарегистрированные организации не могут увидеть чужой лот и сделать ставку —
   маркетплейс невозможен. Проверено живым прогоном 16.07.2026.
   **Решение принято (16.07.2026):** лот виден всем допущенным покупателям в
   обезличенном виде (без продавца, адреса и контактов — против обхода платформы);
   деанонимизация поэтапная: стороны — после создания сделки, контакты — на
   исполнении.
   **Сделано (16.07.2026): витрина.** `GET /api/market/lots` — обезличенная
   read-модель поверх tenant-модели: SQL-функция `market.list_open_lots`
   (SECURITY INVOKER) + отдельная RLS-политика, активная только под
   транзакционным флагом функции; состав колонок зафиксирован на уровне SQL
   (без продавца/адреса/контактов), частичный индекс под федеральный объём,
   пагинация курсором, юнит-тесты. Проверено живьём: покупатель из чужого
   tenant видит ФГИС-лот обезличенно, leak-check чистый.
   **Сделано (16.07.2026): кросс-tenant допуск и ставка.** Миграция
   `20260716150000_auction_cross_tenant_participation`: допуск и ставка ищут лот
   по идентификатору (авторитет — роль оператора и действующий допуск ADMITTED,
   не tenant), снято требование «организация покупателя в tenant'е лота»,
   добавлены политики участника (свои допуски + допущенные лоты). Проверено
   живьём: оператор допустил организацию из чужого tenant (201), её покупатель
   поставил лидирующую ставку (201), витрина показывает ставку обезличенно,
   изоляция сделок не нарушена (deals count 0 у постороннего).
   **Сделано (16.07.2026): рабочий экран торгов кросс-tenant.** `getWorkspace`
   пускает допущенного участника по действующему допуску (дочерние выборки — в
   tenant'е лота), политики участника на ставки/награды добавлены миграцией
   `20260716160000`. Проверено живьём: buyer2 из чужого tenant видит workspace
   (200, лучшая ставка, таймер); same-tenant доступ не изменился. SQL-обход из
   LOCAL_RUNBOOK больше не нужен.
   **Продуктовый вопрос вдогонку:** в workspace лучшая ставка показывает
   buyerName/buyerOrgId других участников — решить, обезличивать ли ставки
   между конкурентами до итога торгов (в витрине уже обезличено).

2. **Нет продуктового пути верификации организации.**
   **Сделано (17.07.2026, API):** контур решения оператора работает штатно.
   `GET /api/organizations?status=PENDING` — очередь на проверку (оператор/
   комплаенс/админ/руководитель); `PATCH /api/organizations/:id/status` —
   решение с обязательным основанием (>=10 символов), валидацией статуса,
   выставлением kycStatus/verifiedAt и записью в audit_events
   (ORGANIZATION_STATUS_DECISION: кто, роль, до/после, основание). Попутно
   починен мёртвый контроллер: легаси JwtAuthGuard читал несуществующий клейм
   role из access-токена — ролевые проверки не работали для боевых токенов;
   переведён на глобальный auth + RolesGuard, список организаций закрыт от
   обычных ролей (раньше любая учётка могла перечислить все организации).
   Проверено живьём: регистрация → PENDING → логин 403 → оператор видит
   очередь → решение с основанием → продавец входит штатно; посторонним 403.
   **Осталось:** экран консоли оператора в web (ядро §3) — API готов.

3. **Нет продуктового пути выдачи операторских ролей.**
   **Сделано (17.07.2026):** `PATCH /api/admin/users/:id/role` — реальная выдача
   (прежний путь менял только synthetic-кэш E2E): только ADMIN, только роли
   операторского контура (SUPPORT_MANAGER/COMPLIANCE_OFFICER/EXECUTIVE/ADMIN),
   обязательное основание, запись ROLE_GRANT_DECISION в audit_events; сессии
   пользователя отзываются триггером БД — роль действует после пере-входа.
   Бутстрап первого администратора — `PC_BOOTSTRAP_ADMIN_EMAIL`, действует
   только пока в системе нет ни одного ADMIN, журналируется от имени SYSTEM.
   Вход ADMIN требует MFA (TOTP) — проверено живьём вместе со всеми негативами
   (чужая роль 409, без основания 409, не-админ 403, старая сессия 401).
   SQL-обходов в LOCAL_RUNBOOK больше не осталось.

## B1 — искажает продуктовый путь

4. **Demo-данные в API-прокси фронта.** Решено (17.07.2026): все встроенные
   демо-массивы и fallback-ветка удалены; каждый путь прокси строго
   real-backend-only с явным `REAL_BACKEND_REQUIRED` (503/401), query string
   форвардится. Тест-контракт запрещает возврат демо-данных. Живой smoke:
   кабинет покупателя работает через очищенный прокси.

5. **`prisma db seed` сломан.** Решено (17.07.2026): битые ссылки на
   несуществующий `prisma/seed.ts` удалены из `apps/api/package.json`;
   единственный сид — `DatabaseSeedService` на старте приложения (описан в
   LOCAL_RUNBOOK).

6. **Сид отгрузок падает на tenant-констрейнте.** Решено (17.07.2026):
   демо-сид `DatabaseSeedService` включается только `SEED_DEMO_DATA=true` или
   в NODE_ENV=test — свежая боевая база не получает фиктивных сделок (§0.1),
   ошибка tenant-констрейнта ушла вместе с самим сидом по умолчанию.

## Проверка госинтеграций (16.07.2026, по запросу владельца)

9. **ФГИС «Зерно» — источник лота работает как заявление, проверка — мок.**
   Аукцион принимает `sourceType: 'FGIS'` с `sourceExternalId` (номер СДИЗ) и
   `sourceCertificateId`; лот регистрируется, источник сохраняется и виден в
   витрине (проверено живьём). Но подтверждение в самой ФГИС — мок:
   `IntegrationsService.pushFgis` возвращает `MOCK_OK` с фиктивным номером СДИЗ,
   вебхук `/api/integrations/fgis/webhook` просто эхо. Нужен честный адаптер по
   §5.2: песочница «оператор подтверждает СДИЗ вручную с журналом» + контракт
   для боевого подключения.
10. **ЕСИА — отсутствует.** `/api/auth/oidc/providers` → `[]`,
    `/api/auth/oidc/authorization-url` → `null`. Регистрация продавца через
    Госуслуги невозможна; добавлен шов §5.4 в конституцию (OIDC-адаптер +
    песочница). SberBusiness OAuth — тоже `not_configured`.
