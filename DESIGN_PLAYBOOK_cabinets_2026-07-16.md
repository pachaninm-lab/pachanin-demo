# Плейбук редизайна личных кабинетов: мировые практики → «Прозрачная Цена»

Дата: 16.07.2026. Продолжение аудита `AUDIT_design-cabinets_2026-07-16.md`.
Задача: понять по лучшим мировым источникам, как сделать ЛК **удобными и понятными — и при этом красивыми, современными и функциональными**, и перевести это в конкретные решения для наших кабинетов.

## 1. Семь законов, на которых сходятся все источники

Каждый закон: **мировая практика → как у нас сейчас → что делаем**.

### Закон 1. Экран отвечает на вопрос пользователя, а не описывает систему
NN/g: дашборд — как приборная панель автомобиля, потребляется быстро и без интерпретации. Polaris: «кратчайший путь дать ровно ту информацию, которая нужна для действия».
**У нас:** кабинет покупателя показывает «контур исполнения», «runtime», «денежный контур» — модель системы, а не вопрос пользователя.
**Делаем:** каждый экран проектируется от одного вопроса. Кабинет: «что мне сделать сейчас?». Карточка сделки: «где моя сделка и что её держит?». Деньги: «где мои деньги и когда придут?». Всё, что не отвечает на вопрос экрана, с экрана уходит.

### Закон 2. Одно решение за раз (GOV.UK)
GOV.UK: «одна вещь на страницу», где вещь — это решение, а не поле; очередь задач со статусами — проверенный паттерн для многошаговых процессов.
**У нас:** до 10 секций-гармошек и 5 «главных» сообщений на первом экране.
**Делаем:** первый экран кабинета — **очередь задач в стиле GOV.UK Task List**: список из 3–7 строк «задача → статус → срок», сгруппированный по сделкам. Клик по задаче — отдельный экран одного решения с одной кнопкой. Никаких гармошек на первом уровне.

### Закон 3. Постепенное раскрытие вместо стены (NN/g Progressive Disclosure)
Раскрытие по мере надобности снижает когнитивную нагрузку до ~55%. Правило: сначала — то, что нужно 80% пользователей в 80% случаев.
**У нас:** гармошки — фиктивное раскрытие: свёрнуто всё, включая главное; раскрыто — повторяется одно и то же 6 раз.
**Делаем:** три слоя данных: (1) статус+действие, (2) контекст по клику «Детали», (3) полный журнал/основания по ссылке «История». Одна цифра живёт в одном месте слоя; повтор суммы на экране = дефект ревью.

### Закон 4. Иерархия — приглушением, а не усилением (Refactoring UI)
Не «сделай главное заметнее», а «сделай второстепенное тише»: вес и цвет шрифта вместо размера, серые подписи, лимит 2–3 веса и 2–3 цвета текста. Сначала макет в сером, цвет — последним.
**У нас:** ALL-CAPS метки над каждым абзацем, всё жирное, красные/оранжевые чипы лентами — усиление всего одновременно.
**Делаем:** типографическая шкала из 4 уровней (display 24 / title 17 / body 15 / caption 13); подписи — обычный регистр, `--ds-color-text-muted`; жирный — только значения и названия объектов; капс запрещён. Цветной фон — только у одного элемента экрана (текущий статус или главное действие).

### Закон 5. Тишина: один баннер, красное — только про деньги/блокер (IBM Carbon, notification UX)
Carbon и вся практика уведомлений: **один баннер на экран**; баннер — для состояния, которое пользователь может разрешить; остальное — инлайн-статусы у места события. «Экран, полный баннеров — коридор кричащих голосов».
**У нас:** 5–6 баннеров и дисклеймеров до первой цифры, «ПРИЧИНА ОСТАНОВКИ» на трёх уровнях.
**Делаем:** жёсткий бюджет: 1 баннер (только разрешимое состояние: «Банк не подтвердил резерв — [Запросить]»), 1 бейдж контекста «Тестовые данные» в шапке, дисклеймеры — один раз в онбординге и футере. Красный — исключительно реальный блокер денег/сделки; жёлтый — ожидание; всё прочее — нейтральное.

### Закон 6. Деньги показываются как в лучших финтехах (Stripe, Wise)
Полная сумма, комиссия и срок — **до** подтверждения; единый экран подтверждения «сумма → получатель → основание → срок»; каждый рубль имеет статус (доступно/в резерве/удержано) и **одну** строку-объяснение; ноль декоративного шума в денежном потоке.
**У нас:** сумма повторена 6 раз, статус денег размазан по кольцу, стрипу, «контуру» и журналу.
**Делаем:** один компонент `MoneyStatus`: три строки (Доступно / В резерве — ждёт банк / Удержано — до акта расхождения), каждая с суммой и одной причиной-ссылкой. Подтверждение действия с деньгами — единый экран со всеми условиями и одной кнопкой. Никаких колец-диаграмм там, где хватает трёх строк.

### Закон 7. Пустой экран продаёт следующий шаг (NN/g Empty States)
Пустое состояние — это онбординг: назвать ценность, показать «форму успеха» (призрачная строка-превью), CTA с глагола.
**У нас:** «0 ₽ — в норме» ×4 и «Сделок пока нет» без пути.
**Делаем:** каждый пустой список = 1 фраза ценности + превью-скелет строки + кнопка-глагол («Создайте закупочный запрос — предложения появятся здесь»). У руководителя пустой портфель показывает, **что появится** и откуда возьмётся.

## 2. Как при этом сделать красиво и современно (2026)

Красота здесь — следствие порядка, а не украшений. Конкретная эстетическая система:

**Сетка (bento как ИА-инструмент).** Первый экран кабинета — bento-композиция из 3 зон с разным весом: большая карточка «Сейчас» (текущая задача №1), средняя «Очередь» (остальные задачи), узкая «Пульс» (деньги/статусы). Не 10 одинаковых карточек в столбик, а модули разного размера по важности — это и есть современный вид без единого украшения.

**Поверхности и глубина.** Один уровень тени для карточек (`--ds-shadow-surface`), один для оверлеев. Радиус 10/16. Фон канвы чуть темнее поверхностей — глубина за счёт слоёв, не за счёт рамок; убрать двойные рамки+тени.

**Цвет.** База нейтральная (канва, поверхность, 3 уровня текста) + фирменный зелёный **только** для первичного действия и подтверждённых состояний + 3 семантических (жёлтый/красный/синий) только по назначению. Правило Stripe: акцент — дозированно, тогда он работает. Тёмная тема — из тех же токенов (уже есть в `--ds-*`).

**Типографика.** Inter для всего, JetBrains Mono — для сумм и номеров объектов (DL-9106, 9 650 000 ₽) — табличные цифры выравниваются, деньги мгновенно отличимы. Это дешёвый приём «дорогого» вида из финтеха.

**Движение (Carbon Productive motion).** Микропереходы 120–200мс на hover/появление строк; никаких каруселей и пульсаций. Скелетоны вместо спиннеров.

**Плотность (Pencil & Paper).** Списки — строки, не карточки-простыни: высота строки 44–56px, переключатель плотности для операторов, фиксированная шапка таблиц, ~25 строк на страницу, действия — в конце строки.

## 3. Рецепт по экранам

### 3.1 Первый экран кабинета (все роли)
```
┌ Шапка: роль · организация · поиск · [Тестовые данные] ┐
├───────────────────────────┬───────────────────────────┤
│  СЕЙЧАС (bento-XL)        │  ДЕНЬГИ (bento-S)         │
│  Подтвердить резерв       │  Доступно      0 ₽        │
│  DL-9106 · до 18:00       │  В резерве  9,65 млн ₽ →  │
│  Без этого не стартует    │  Удержано   624 тыс. ₽ →  │
│  логистика                ├───────────────────────────┤
│  [Запросить у банка]      │  СОБЫТИЯ (bento-S)        │
│  Подробнее →              │  10:24 Акт готовится      │
├───────────────────────────┤  09:12 Ставка принята     │
│  ОЧЕРЕДЬ (task list)      │  Вся история →            │
│  ○ Закрыть расхождение    │                           │
│    веса · LOT-2403 · акт  │                           │
│  ○ Ответить на предложение│                           │
│  ✓ Допуск подтверждён     │                           │
└───────────────────────────┴───────────────────────────┘
```
Одна задача-герой, очередь по GOV.UK, деньги тремя строками, события — тихая лента. Больше на первом экране нет ничего.

### 3.2 Экран решения (клик по задаче)
Один вопрос — один экран (GOV.UK): заголовок-глагол («Подтвердить фактический вес 598,8 т»), 3–5 фактов, что изменится после действия (fintech-подтверждение: сумма/срок/последствие), одна первичная кнопка + «Назад». Основания и журнал — ссылкой вниз.

### 3.3 Карточка объекта (сделка/рейс/партия)
Слева таймлайн событий (что произошло → кто → что открылось), справа — липкая сводка: этап, сумма `MoneyStatus`, ответственный, следующий шаг. Вкладки: Обзор · Документы · Деньги · Качество. Это замена восьми гармошек.

### 3.4 Полевой режим (водитель, приёмка)
По мировой практике полевых приложений: 3–4 пункта навигации максимум; одна задача на весь экран; кнопки от 48px (перчатки, солнце); статус — цветом и словом, не абзацем; постоянно видимый индикатор синка («Сохранено на устройстве · отправится при связи»); подтверждение прибытия/веса — одно касание + камера. Убрать с полевого экрана футер с интеграциями и юридические блоки полностью.

### 3.5 Руководитель
Calm-режим: одна строка-вердикт («Всё в норме · 3 сделки идут по плану») вместо четырёх нулевых карточек; ниже — только исключения (SLA, споры, удержания). Пустой портфель — превью того, что появится.

## 4. Definition of Done для любого нового экрана ЛК

- [ ] Назван вопрос пользователя, на который отвечает экран (одна фраза в описании PR)
- [ ] Одно первичное действие; вторичные — визуально тише (Refactoring UI)
- [ ] ≤ 1 баннера; дисклеймеров в теле экрана — 0 (Carbon)
- [ ] Ни одна цифра не повторяется на экране дважды
- [ ] Капс-метки: 0; уровней шрифта: ≤ 4; цветов текста: ≤ 3
- [ ] Красный присутствует только при реальном блокере
- [ ] Пустое состояние: ценность + превью + глагол-CTA (NN/g)
- [ ] Все стили — из `--ds-*`; новых hex: 0 (гейт `design:v8:guard`)
- [ ] Мобильная версия: цели ≥ 44px, ничего не перекрывает контент, высота ленты ≤ ~1500px
- [ ] Скелетон на загрузке; ошибка данных не съедает экран (показываем последнее известное состояние + строку «не обновилось»)

## 5. Чем это отличается от прошлой версии концепции

Прошлый документ предлагал «очередь решений» из первых принципов. Этот — приземляет её на проверенные образцы: очередь = GOV.UK Task List (протестирован государством на реальных пользователях), экран решения = One Thing Per Page, деньги = финтех-паттерн Stripe/Wise, тишина = нотификационный паттерн IBM Carbon, иерархия = Refactoring UI, полевой режим = стандарты логистических приложений, эстетика = bento/calm 2026 + «спокойная технология» Stripe. То есть каждый элемент нового ЛК имеет мировой прецедент, на который можно сослаться при защите решения.
